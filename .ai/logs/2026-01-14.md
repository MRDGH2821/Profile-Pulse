### Architecture Decisions

**Why Vec<String> for emails/phones/URLs?**
- Simple, dynamic array structure
- Easy add/remove operations
- Indexed message system maps naturally
- First non-empty value becomes primary field
- Additional values stored in custom_fields

**Why custom_fields HashMap?**
- Extensible for future field types
- Compatible with VCF arbitrary properties
- Avoids database schema changes for new fields
- Matches existing Contact model design

**Why alphabetical pagination over simple pagination?**
- More natural for large contact lists (think phone book)
- Faster to find specific contact when you know the name
- Reduces cognitive load (jump to "M" instead of guessing page number)
- Common UX pattern in contact management apps

**Why sub-pagination within letters?**
- Handles edge cases (100+ contacts starting with "S")
- Keeps UI responsive with very large lists
- 50 items per page is reasonable scroll depth
